Runtime Flows

Role Radar 用户请求链路与后台任务链路

这份文档说明当前生产架构里，哪些操作只会读本地数据库，哪些操作会调用 Brave Search、Gemini、MiniMax、Turnstile、SMTP。重点回答一个常见问题： 用户只是浏览网页时，会不会额外产生外部搜索或大模型成本。

适用版本：v2.1 数据库：SQLite 外部搜索：Brave 模型：Gemini / MiniMax
用户访问链路

默认是只读本地数据

首页 / 岗位详情 / 中英文切换

这些请求默认只会命中 Next.js 服务端和 SQLite，不会临时去调 Brave 搜索，也不会临时去调 Gemini 或 MiniMax。

watchlist 登录与订阅

这条链路会命中站内 API，并可能调用 Turnstile 和 SMTP，但不会触发 Brave 搜索，也不会跑岗位时间线归因。

页面展示的时间线

页面只读取已经落库的 Signal 和 RoleEvidenceCandidate，不会在用户点击岗位时现搜现算。

后台任务链路

外部成本主要在这里产生

npm run ingest:signals

负责源抓取、归一化、岗位归因、风险刷新。这条链路可能调用 Gemini 或 MiniMax。

npm run discovery:roles

负责按岗位搜索补洞。会调 Brave Search；遇到规则不够确定的候选时，可能再调 MiniMax 或 Gemini 做 adjudication。

scripts/run-daily-cycle.sh

daily scheduler 的入口。是否执行岗位搜索补洞，取决于 ROLE_RADAR_DAILY_ENABLE_ROLE_DISCOVERY。

时间线为什么会有空岗位
  • 当前页面不会即时搜索，所以岗位页只展示已经缓存和落库的时间线。
  • daily discovery 默认不是全量跑 700 多个岗位，而是按 limit 跑一小批稀疏岗位。
  • 即使搜到了结果，也还要经过过滤和归因，landing page、职业百科、弱相关页面会被挡掉。
  • 因此“当前没有时间线”不代表永远没有，只代表这轮还没补到，或补到后被过滤掉了。
当前缓存与重试规则
  • 岗位搜索结果会写入 SearchQueryRun 和 SearchHit。
  • 当前单次 query cache 默认新鲜期是 24 小时。
  • 后续再跑 discovery 时，过期了会重新搜；没过期则优先复用缓存。
  • 最终只有 timelineEligible=true 的证据会出现在岗位时间线中。
运维建议
如果你希望生产环境持续补时间线

把 ROLE_RADAR_DAILY_ENABLE_ROLE_DISCOVERY 设为 1，并把 ROLE_RADAR_DAILY_ROLE_DISCOVERY_LIMIT 设为一个你能接受成本的值，例如 20。如果只是用户正常浏览网页，不需要担心 Brave 或大模型请求被放大；真正的成本主要来自 daily/manual 的后台任务。
